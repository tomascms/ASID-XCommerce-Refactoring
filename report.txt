## 🎯 Visão Geral

### Contexto da Transformação

O projeto XCommerce passou por uma transformação profunda de arquitetura:
- **De**: Uma aplicação monolítica única (Spring Boot 2.1.9, Java 8)
- **Para**: Uma arquitetura de microserviços distribuída (Spring Boot 3.4.0, Java 21)

Esta transformação não foi apenas uma mudança de código, mas uma **rearchitetura completa do sistema**, envolvendo:
- Decomposição funcional
- Separação de dados
- Implementação de padrões distribuídos
- Modernização de tecnologias
- Adição de camadas de observabilidade e resiliência

## 🏛️ Arquitetura do Monolito

### Estrutura Geral

```
XCommerce Monolith (Spring Boot 2.1.9)
│
├── Aplicação Única (xcommerce-monolithic-master)
│   ├── Controllers (REST)
│   ├── Services
│   ├── Repositories (JPA)
│   ├── Security (JWT + JDBC Authentication)
│   └── Data Models
│
├── Infraestrutura:
│   ├── 1x PostgreSQL (Single Database)
│   ├── 1x Redis (Caching)
│   └── 1x Application Server
```

### Características Principais

#### **1. Stack Tecnológico (Monolito)**

```properties
Java Version:              1.8
Spring Boot:              2.1.9.RELEASE
Spring Security:          Built-in (JDBC Authentication)
Caching:                  Redis
Database:                 PostgreSQL (1 único banco de dados)
API Documentation:        N/A (sem OpenAPI/Swagger)
Message Broker:           N/A (sem event-stream)
Observability:            Logs apenas
Containerization:         N/A (tradicional)
Service Communication:    N/A (monolítico)
```

#### **2. Arquitetura Logística**

O monolito segue o padrão **MVC-SOA**:

```
HTTP Request
    ↓
API Gateway Implícito (Spring DispatcherServlet)
    ↓
Controllers (REST Endpoints)
    ↓
Service Layer (Business Logic)
    ↓
Repository Layer (JPA/Data Access)
    ↓
PostgreSQL Database (SINGLE)
    ↓
Response
```

#### **3. Modelo de Dados**

**Um único banco de dados com todas as tabelas:**

```sql
-- Tabelas de Domínio
CREATE TABLE "user" (...)
CREATE TABLE authority (...)
CREATE TABLE product (...)
CREATE TABLE brand (...)
CREATE TABLE category (...)
CREATE TABLE cart (...)
CREATE TABLE "order" (...)
CREATE TABLE order_item (...)
CREATE TABLE inventory (...)
CREATE TABLE payment (...)
CREATE TABLE notification (...)
```

**Características:**
- ✅ ACID garantido (transações monolíticas)
- ✅ Integridade referencial forte
- ❌ Escalabilidade limitada (gargalo único)
- ❌ Contato direto entre tabelas

#### **4. Segurança (Monolito)**

```java
@EnableWebSecurity
public class SecurityConfigurer extends WebSecurityConfigurerAdapter {
    
    // Autenticação via JDBC direto
    authenticationManagerBuilder.jdbcAuthentication()
        .dataSource(dataSource)
        .usersByUsernameQuery("SELECT username, password, active FROM user ...")
        .authoritiesByUsernameQuery("SELECT username, authority FROM authority ...");
    
    // Autorização via Roles
    .authorizeRequests()
        .antMatchers("/rest/user/admin").hasRole("SUPERADMIN")
        .antMatchers("/rest/catalog/**").hasRole("ADMIN")
    
    // JWT + Session Stateless
    .addFilter(new JwtInterceptingFilter())
    .sessionCreationPolicy(SessionCreationPolicy.STATELESS);
}
```

#### **5. Funcionalidades**

Integradas no **único código executável**:

- ✅ Autenticação e Autorização
- ✅ Gestão de Catálogo (Produtos, Marcas, Categorias)
- ✅ Shopping Cart
- ✅ Gestão de Pedidos
- ✅ Gestão de Inventário
- ✅ Processamento de Pagamentos
- ✅ Sistema de Notificações
- ✅ Gestão de Utilizadores

```
├── Services: AuthService

└── BD: xcommerce_auth
```

##### **📚 Catalog Service (8082)**
- **Responsabilidade**: Gestão de Produtos, Marcas, Categorias
- **Endpoints**: 
  - `/products/**` (CRUD de produtos)

  - `/categories/**` (CRUD de categorias)
- **Eventos Kafka**: Publica `product-events`
- **BD Própria**: `xcommerce_catalog`
- **Resiliência**: Circuit Breaker + Retry + Bulkhead

```
Catalog Service
```

**Implementação (Order Service):**
```java
@Service
public class OrderService {
    @Autowired private InventoryClient inventoryClient;
    @Autowired private CartClient cartClient;
    @Autowired private CatalogClient catalogClient;
    
    public Order createOrder(String userId) {
        // 1. Validar carrinho
        List<CartItem> items = cartClient.getItems(userId);
        
        // 2. Validar stock
        for (CartItem item : items) {
            if (!inventoryClient.isAvailable(item.getProductId())) {
                throw new OutOfStockException();
            }
        }
        
        // 3. Reservar stock
        ReservationRequest request = new ReservationRequest();
        request.setItems(items);
        inventoryClient.reserve(request);
        
        // 4. Criar ordem
        Order order = new Order();
        order.setUserId(userId);
        order.setStatus(OrderStatus.CREATED);
        return orderRepository.save(order);
    }
}
```

```java
public class OrderService {
    @Autowired private KafkaTemplate<String, String> kafkaTemplate;
    
    public void placeOrder(Order order) {

        );
    }

// 2. Payment Service escuta e processa
@Service
public class PaymentProcessor {
    @Autowired private KafkaTemplate<String, String> kafkaTemplate;
    
    @KafkaListener(topics = "order-placed-events", groupId = "payment-group")
    public void processOrderPayment(String orderId, String message) {

            kafkaTemplate.send("payment-events", 
                orderId, 
                "PAYMENT_FAILED:" + e.getMessage()
            );
        }
    }
}
```

#### **4. Estratégia de Dados (Polyglot Persistence)**

**Monolito:**
```
1 PostgreSQL Instance
        ↓
  [SINGLE DATABASE]
        ↓
9 Tabelas (todas acopladas)
```

**Microserviços:**
```
1 PostgreSQL Instance (9 Bases Separadas)
        ↓
┌─────┬─────┬─────┬─────┬─────┬─────┬─────┬─────┬─────┐
│auth │catalog│cart │orders│inventory│users│payment│notification│keycloak│
└─────┴─────┴─────┴─────┴─────┴─────┴─────┴─────┴─────┘
        ↓       ↓       ↓       ↓       ↓       ↓       ↓       ↓
    [Serviço Autónomo com Schema Próprio]
```

**Benefícios:**
- ✅ Escalabilidade independente (cada BD pode ser replicada)
- ✅ Sem acoplamento via referências cruzadas
- ✅ Flexibilidade: serviço pode mudar de engine (PostgreSQL → MongoDB, etc.)
- ✅ Migrações independentes
- ✅ Isolamento de falhas (crash de um não afeta outro)

#### **5. Resiliência e Tolerância a Falhas**

**Resilience4j Implementado em Cada Serviço:**

```java
@Configuration
public class ResilienceConfig {
    
    // Circuit Breaker: protege contra cascata de falhas
    @Bean
    CircuitBreakerRegistry circuitBreakerRegistry(MeterRegistry meterRegistry) {
        CircuitBreakerConfig config = CircuitBreakerConfig.custom()
            .failureRateThreshold(50)              // Abrir se 50% falhas
            .minimumNumberOfCalls(4)               // Mínimo 4 chamadas antes de decidir
            .slidingWindowSize(8)                  // Janela de 8 chamadas
            .waitDurationInOpenState(Duration.ofSeconds(10))  // Tentar após 10s
            .permittedNumberOfCallsInHalfOpenState(2)         // 2 chamadas teste
            .slowCallDurationThreshold(Duration.ofSeconds(2)) // >2s é lento
            .slowCallRateThreshold(75)             // Se 75% lentos, abrir
            .build();
        
        CircuitBreakerRegistry registry = CircuitBreakerRegistry.of(config);
        TaggedCircuitBreakerMetrics.ofCircuitBreakerRegistry(registry)
            .bindTo(meterRegistry);
        return registry;
    }
    
    // Retry: tenta novamente com backoff
    @Bean
    RetryRegistry retryRegistry(MeterRegistry meterRegistry) {
        RetryConfig config = RetryConfig.custom()
            .maxAttempts(3)                           // 3 tentativas
            .waitDuration(Duration.ofMillis(500))     // Esperar 500ms entre tentativas
            .failAfterMaxAttempts(true)               // Falhar após 3 tentativas
            .build();
        
        RetryRegistry registry = RetryRegistry.of(config);
        TaggedRetryMetrics.ofRetryRegistry(registry).bindTo(meterRegistry);
        return registry;
    }
    
    // Bulkhead: isola threads para evitar resource exhaustion
    @Bean
    BulkheadRegistry bulkheadRegistry(MeterRegistry meterRegistry) {
        BulkheadConfig config = BulkheadConfig.custom()
            .maxConcurrentCalls(10)                    // 10 chamadas paralelas máx
            .maxWaitDuration(Duration.ofMillis(250))   // Esperar 250ms max na fila
            .build();
        
        BulkheadRegistry registry = BulkheadRegistry.of(config);
        TaggedBulkheadMetrics.ofBulkheadRegistry(registry).bindTo(meterRegistry);
        return registry;
    }
}
```

**Degradação Graciosa no API Gateway:**

```java
@Bean
public RouteLocator gatewayRoutes(RouteLocatorBuilder builder) {
    return builder.routes()
        .route("order-service", r -> r.path("/order/**")
            .filters(f -> f.circuitBreaker(c -> c
                .setName("order-gateway-cb")
                .setFallbackUri("forward:/fallback/order")  // ← degradação
            ))
            .uri("http://order-service:8084"))
        .build();
```

```yaml
    KC_DB_URL: jdbc:postgresql://xcommerce-db:5432/xcommerce_keycloak
    KC_HOSTNAME: keycloak
    KC_PROXY: edge
```

**JWT + OAuth 2.0:**
- Cada serviço valida tokens JWT emitidos pelo Keycloak
- Não há compartilhamento de senhas entre serviços
- Revogação centralizada

**Isolamento de Rede:**
- Todos os serviços na mesma rede Docker (`xcommerce-network`)
- Sem exposição direta ao exterior (apenas Gateway)
- Comunicação intra-rede privada

### **Padrão de Melhoria 2: De Transações ACID para Saga Pattern**

```
MONOLITO (Consistência Imediata)
    
CREATE_ORDER:
    BEGIN;
        INSERT INTO orders ...;
        UPDATE inventory SET qty = qty - 1 ...;
        INSERT INTO payment ...;
        UPDATE notifications ...;
    COMMIT;  // Atomic: tudo ou nada
    
Problema: Se 1 serviço falha, tudo falha

MICROSERVIÇOS (Eventual Consistency com Saga)

CREATE_ORDER:
    1. Order Service cria ordem (isolada)
       └─→ PublishEvent("order.created")
    
    2. Inventory Service consome evento
       └─→ UpdateStock() ou PublishEvent("order.inventory_failed")
    
    3. Payment Service consome evento
       └─→ ProcessPayment() ou PublishEvent("order.payment_failed")
    
    4. Se falhar:
       └─→ Compensating transactions (refund, restore stock)
    
Benefício: Resiliente a falhas parciais
```

```
MONOLITO (Deployment Raro)
```

```
Até mesmo: Auto-scaling pode ativar automaticamente!
```

## 🎓 Conclusões

### **Por Que XCommerce Precisava de Microserviços?**

#### **1. Escalabilidade**
- ❌ **Monolito**: 1 JVM = 1 limite de recursos
- ✅ **Microserviços**: N JVMs = Escalabilidade N vezes

#### **2. Agilidade**
- ❌ **Monolito**: Deploy semanal (risco)
- ✅ **Microserviços**: Deploy diário (seguro)

#### **3. Resiliência**
- ❌ **Monolito**: 1 bug = Sistema inteiro afetado
- ✅ **Microserviços**: 1 bug = Serviço isolado + Fallback

#### **4. Flexibilidade**
- ❌ **Monolito**: Stack fixo (Java 1.8 para sempre?)
- ✅ **Microserviços**: Cada serviço sua stack

#### **5. Observabilidade**
- ❌ **Monolito**: Logs e esperança
- ✅ **Microserviços**: Tracing distribuído + Métricas + Dashboards

### **Quando o Monolito Era Apropriado?**

```
Monolito é ÓTIMO para:
├── MVP (Quick to market)
├── Aplicações pequenas (<50KB de código)
├── Single tenant
├── Carga previsível
└── Equipa pequena (<5 pessoas)

DEPOIS crescer para Microserviços
```

### **Trade-offs da Transformação XCommerce**

| O que Ganhou | O que Perdeu |
|---------|---------|
| ✅ Escalabilidade independente | ⚠️ Latência de rede (+40ms) |
| ✅ Deployment independente | ⚠️ Complexidade operacional |
| ✅ Resiliência melhorada | ⚠️ Debugging distribuído difícil |
| ✅ Flexibility tecnológica | ⚠️ Testing mais complexo |
| ✅ Equipes independentes | ⚠️ DevOps overhead |
| ✅ Eventual transparency | ⚠️ Consistência eventual |

### **Lições Aprendidas**

1. **Timing é tudo**
   - Não fazer microserviços no dia 1 (monolito MVP é mais rápido)
   - Se aplicação crescer, fazer a transformação

2. **DevOps é crítico**
   - Microserviços sem DevOps = caos
   - Container orchestration (Docker/Kubernetes) é obrigatório
   - Monitoring automática (Prometheus/Grafana) é essencial

3. **Dados são o desafio**
   - Separar dados = Problema difícil
   - Eventual consistency é contraprodutiva para algumas operações
   - Saga pattern = Solução elegante mas complexa

4. **Testing muda**
   - Testes unitários: Rápidos ✅
   - Testes de integração: Entre serviços = Lento
   - Contract testing: Solução (mock via Pact)

5. **Segurança centralizada**
   - Keycloak = Correção sábia
   - OAuth 2.0 = Padrão correto
   - JWT para autenticação entre serviços
