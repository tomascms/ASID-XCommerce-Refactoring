🏛️ Enterprise Architecture & Microservices Documentation

📈 System Architecture Overview

Type: Event-Driven Microservices with Distributed API Gateway

Services: 9 independent microservices + 1 API Gateway + Admin Dashboard

Communication: Synchronous (REST/HTTP) + Asynchronous (Apache Kafka)

Data Model: Database-per-service (polyglot persistence)

Stack: Spring Boot 3.4, Java 21, PostgreSQL, Apache Kafka, Docker, Express.js Dashboard

🔧 Complete Microservice Catalog

🚪 API Gateway (Port 9000)

Framework: Spring Cloud Gateway

Purpose: Single reverse proxy entry point. Authenticates all requests, routes to services, handles cross-cutting concerns.

Responsibilities:

  • JWT token validation & refresh
  • Request/response routing (9 microservices)
  • CORS & security headers
  • Rate limiting & throttling
  • Circuit breaker integration
  • Request logging & tracing

📊 Load: Handles aggregation of requests from Dashboard + clients
⚡ Performance: ~50ms latency per request

📦 Catalog Service (Port 8081)

Framework: Spring Boot + Spring Data JPA + REST

Database: PostgreSQL (catalog_db - dedicated)

Purpose: Complete product catalog management (products, brands, categories)

API Endpoints:

  • GET /products (paginated, filterable)
  • POST /products (create with validation)
  • PUT /products/{id} (update entire product)
  • DELETE /products/{id}
  • GET /brands, PUT /brands, DELETE /brands
  • GET /categories, PUT /categories, DELETE /categories

Events Published: product-events (Kafka)
Cache: 1-hour TTL on products list
Partitions by: Brand, Category (efficient querying)

📊 Inventory Service (Port 8082)

Framework: Spring Boot + Pessimistic Locking

Database: PostgreSQL (inventory_db - dedicated)

Purpose: Real-time stock management with distributed lock pattern

API Endpoints:

  • GET /inventory/{productId} - Get stock
  • POST /inventory/decrease - Atomic decrement (with retry)
  • POST /inventory/increase - Atomic increment
  • PUT /inventory/{id} - Update record

Critical Path: Order received → Inventory locked → Stock checked → Decremented
Retry Strategy: Exponential backoff (1s, 2s, 4s, 8s max)
Transactions: Pessimistic locking prevents race conditions

📋 Order Service (Port 8083)

Framework: Spring Boot + Saga Pattern

Database: PostgreSQL (order_db - dedicated)

Purpose: Distributed order orchestration using Saga pattern

API Endpoints:

  • POST /orders - Initiate saga: Order → Inventory → Payment → Confirm
  • GET /orders/{id}
  • PUT /orders/{id}/status
  • GET /orders/user/{userId}

Saga Flow:
1️⃣ Create order (PENDING_PAYMENT)
2️⃣ Call Inventory Service (sync) - if fails, rollback
3️⃣ Call Payment Service (sync) - if fails, restore inventory (compensate)
4️⃣ Publish order-placed-event to Kafka
5️⃣ Return CONFIRMED to client

💳 Payment Service (Port 8084)

Framework: Spring Boot + External Gateway Integration Pattern

Database: PostgreSQL (payment_db - dedicated)

Purpose: Payment processing with idempotency & transaction logging

API Endpoints:

  • POST /payments/process - Charge customer (idempotent)
  • GET /payments/{id}
  • PUT /payments/{id}/status
  • POST /payments/{id}/refund

Pattern: Idempotent key prevents duplicate charges
Events: Publishes payment-events (success/failure)
Simulation: 95% success rate for realistic testing

📧 Notification Service (Port 8085)

Framework: Spring Boot + Kafka Consumer Groups

Purpose: Asynchronous event consumption & notification dispatch

Kafka Topics Subscribed:

  • user-events → 📧 Login alerts
  • product-events → 📢 New product newsletters
  • order-placed-events → 📦 Order confirmations
  • payment-events → 💳 Payment success/failure alerts
  • order-confirmed-events → ✅ Confirmation emails
  • order-cancelled-events → ❌ Cancellation notices

Pattern: Consumer groups for scalable processing
Output: Console logs + email simulation
Scalability: Add instances = automatic partition rebalancing

🔄 Communication Patterns

Synchronous (REST/HTTP)

Use: When immediate response required

Dashboard → API Gateway → Order Service → Inventory Service (waits for response)

✅ Pros: Immediate feedback, simple
❌ Cons: Tight coupling, availability dependency

Retry Config: Max 3 retries, 1s-8s exponential backoff

Asynchronous (Kafka Events)

Use: When response not immediately needed

Order Service → Publishes event → Notification Service consumes (async)

✅ Pros: Loose coupling, scalable, resilient
❌ Cons: Eventually consistent

Throughput: 18 partitions × 1000 msg/s = 18K msg/s capacity

🛡️ Fault Tolerance & Resilience Patterns

🔁 Retry Logic

Algorithm: Exponential backoff with jitter

  • Attempt 1: immediate
  • Attempt 2: 1s wait
  • Attempt 3: 2s wait
  • Attempt 4: 4s wait (max)

⚡ Circuit Breaker

States: CLOSED → OPEN → HALF_OPEN

Stops calling failing service after 50% error rate for 30s

⏱️ Timeouts

HTTP Timeout: 5 seconds

Prevents requests hanging, triggers fallback/retry

🚪 Bulkhead

Pattern: Thread pool isolation

One slow service doesn't starve thread pool

💾 Fallback

Strategy: Return cached data

If service down, serve stale but usable response

🔍 Health Checks

Endpoint: GET /actuator/health

Checks DB, Kafka, API dependencies

🗄️ Data Persistence & Consistency

Database-Per-Service Architecture

Principle: Each microservice owns its data. No shared databases = independent scaling.

catalog_db
Products, Brands, Categories
inventory_db
Stock levels, reservations
order_db
Orders, order items
payment_db
Transactions, receipts

✅ Benefits: Independent scaling, fault isolation, technology choice per service, no central bottleneck

Data Consistency Model: Eventual Consistency (BASE)

Principle: Systems eventually reach consistent state (not immediately like ACID)

Flow Examples:

Scenario 1: Order Created
T0: Order Service creates order (status=PENDING_PAYMENT) ← Local ACID transaction
T1: Inventory Service reduces stock ← Separate ACID transaction
T2: Payment Service processes payment ← Separate ACID transaction
T3: system consistent across all services ✅

Scenario 2: Payment Fails
T0: Order created
T1: Stock reduced
T2: Payment REJECTED
T3: Order Service compensates (rollback):
     - Inventory Service: Restore stock
     - Order Service: Set status=CANCELLED
T4: system eventually consistent after compensation ✅

Pattern Used: Saga Pattern (Choreography) - events trigger next step
Guarantee: Either all steps succeed OR all compensate (no partial state)

📈 Scalability & Performance Engineering

Horizontal Scaling

Model: Stateless services → add instances behind load balancer

Scenario: Black Friday = 10x traffic spike
Action: Scale Order Service 1 → 10 instances
Result: Handle 10,000 req/sec vs 1,000 req/sec
Other services: unchanged (users demand catalog info, not orders)

✅ Independent scaling = cost efficient (no unnecessary resource waste)

Kafka Partitioning & Consumer Groups

Architecture: 6 topics × 3 partitions = 18 partitions

Topic: order-placed-events
├─ Partition 0 ← Notification Service Instance 1
├─ Partition 1 ← Notification Service Instance 2
└─ Partition 2 ← Notification Service Instance 3

When add 4th instance:
Kafka rebalances → 4th instance takes over Partition (round-robin)

Throughput: Each partition ~1000 msg/s = 18K msg/s total
Scaling: Add 3 more notification instances → 36 partitions → 36K msg/s (linear)

Caching Strategy (Multi-Layer)

Layer 1: Application cache (Spring Cache) - Catalog products cached 1 hour
Layer 2: Database indexes - B-tree on frequent queries
Layer 3: Connection pooling - Reuse DB connections (HikariCP)
Layer 4: Distributed cache - Redis for multi-instance scenarios (optional)

📊 Monolith vs Microservices: Business Impact

Aspect ❌ Monolith ✅ Microservices
Deployment All or nothing (5 min restart) Individual service (0 downtime)
Scaling Scale entire app (40% wasted) Scale only what's needed (60% savings)
Resilience Bug = whole system down Fault isolated, system continues
Team Work Coordination hell, merge conflicts Independent teams, own service
Tech Stack Forever locked (Java monolith = forever Java) Mix: Notification could be Go, Catalog stay Java
Performance No network latency (~0ms calls) Network overhead (50ms) but parallelizes
Complexity Simple at start, quickly becomes spaghetti Complex upfront, scales cleanly
Consistency ACID (single transaction) Eventual (Saga pattern)
